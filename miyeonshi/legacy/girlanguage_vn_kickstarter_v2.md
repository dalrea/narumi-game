## 1) 캐릭터 설정 (Character Bible)

### 1.1 히로인: 미나 (Mina)
- **나이**: 24 (대학교 졸업 유예/취준 중, 혹은 대학원 진학 준비 중 택1)
- **관계**: 동아리 운영진. “나”는 같은 동아리의 실무 담당(신입 운영진/후배 느낌)
- **외형 키워드(텍스트 기반 묘사)**  
  - 단정한 캐주얼(셔츠+가디건), 손목시계, 머리카락은 어깨선 정도  
  - 감정이 올라오면 **말보다 표정/호흡이 먼저 변함**(시선 회피, 미소 고정, 숨 길어짐)
- **성격 요약(플레이에 반영되는 트리거)**
  - 자존심 강함(질투/서운함을 “확인”으로 포장)
  - 책임감 높음(일이 꼬이면 관계보다 일을 먼저 처리)
  - 감정 표현은 간접적이지만, **일단 신뢰가 쌓이면 급격히 직설이 됨**
- **직설 못하는 이유(구체화)**
  - 과거에 감정을 솔직히 말했는데 “별 거 아닌 걸로 예민하다”는 말을 들어 **한 번 크게 데인 경험**  
  - 이후로 감정을 **질문/농담/완곡어**로 감싼다.  
  - 단, 상대가 “기억하고 행동으로 증명”하면 방어가 풀린다.
- **말투 패턴(시나리오 작성 규칙)**
  1) 불편할 때: “아니야~ 괜찮아ㅎㅎ”, “그냥 그렇다고”, “됐어.”  
  2) 기대할 때: “너 요즘… (멈춤)”, “아무거나~ (근데 표정으로 제외항목 제시)”  
  3) 상처받았을 때: **짧아짐**(단답+이모티콘 감소)  
  4) 신뢰할 때: **주어가 늘어남**(“나는 사실…”, “내가 좀…”)  
- **좋아하는 것(컨텐츠 소재)**
  - 편의점 신상 간식, 예쁜 문구류, 사소한 “기억해주는 행동”
- **싫어하는 것(갈등 소재)**
  - “말 안 하면 모른다”를 무기처럼 던지는 태도(해결책처럼 보이지만 사실 회피)
  - 약속/연락 패턴이 흐트러지는 것(관계의 안전감 저하)

### 1.2 주인공: “나” (Player)
- **나이**: 23
- **포지션**: 동아리 실무 담당(행사/공지/대외 연락). 일은 성실하지만 관계 신호 해석이 흔들림.
- **성격**
  - 착함/성실함이 무기지만, “정답 찾기”에 집착하는 타입  
  - 눈치가 흔들리는 이유:  
    - 사람 기분을 맞추려다 **오히려 질문을 과도하게 던짐**(피로도 증가)  
    - 실패 경험(과거 썸에서 “너 너무 계산적이야” 소리 들음) → **직설/눈치 사이에서 오락가락**
- **대사 톤**
  - 마음속 독백이 많고, 말은 깔끔하게 하려다 어색해짐  
  - 실수하면 “설명”으로 만회하려는 경향(미나에게 피곤함 유발 가능)

### 1.3 조연(선택/짧게)
- **동아리 회장(조연1)**: 일 위주, 갈등이 생기면 “그냥 말로 하자”를 자주 씀 → 미나의 트라우마 자극 장치
- **친구(조연2)**: 밈 과몰입 조언자(“그거 여자어잖아”) → 밈 맹신의 함정 장치

## 2) 챕터별 시놉시스 (20~30분 확보용)

> 각 챕터는 **장소/시간/핵심 갈등/클라이맥스 선택/로그 테마**를 고정해두면 Claude가 흔들리지 않습니다.

### Ch1. 프롤로그: 괜찮아의 온도
- **시간/장소**: 평일 저녁, 동아리방/복도
- **핵심 갈등**: 늦은 합류 + 미나의 “괜찮아ㅎㅎ”가 진짜인지 확인
- **클라이맥스 선택**:  
  - “당장 해결하려 든다” vs “기억하고 행동으로 챙긴다”
- **로그 테마**: ‘괜찮아’는 종결이 아니라 신호 / 웃고 넘기기의 비용

### Ch2. 친해짐: 아무거나의 반경
- **시간/장소**: 주말 오후, 카페 → 밥집
- **핵심 갈등**: “아무거나”를 정답 게임으로 만들면 피로가 쌓임.  
  - **밈대로 과해석하면 실패**하는 구간 필수 삽입: 미나가 진짜 아무거나인 순간.
- **클라이맥스 선택**:
  - 메뉴 결정 방식(옵션 제시, 선호 확인, 랜덤, 추천 강요 등)
- **로그 테마**: “선택 피로” / “제안은 배려지만 강요가 되기도”

### Ch3. 미묘함: 바빠
- **시간/장소**: 평일 밤, 메신저(채팅 화면 연출) + 다음날 캠퍼스
- **핵심 갈등**: 미나의 “바빠”가 **단순 일정**인지 **관계 우선순위 조정**인지.  
  - 주인공은 불안해져서 설명/질문을 과다하게 할 위험.
- **클라이맥스 선택**:
  - “공간을 준다” vs “확인하려 든다” vs “구체적 제안(언제/어디)”
- **로그 테마**: “바쁘다”의 의미는 문장보다 패턴(반복/시간대/말끝)

### Ch4. 갈등: 됐어
- **시간/장소**: 동아리 행사 당일, 현장(소란/바쁨) → 행사 후 귀가길
- **핵심 갈등**: 누적 피로/서운함이 폭발. “됐어”는 **대화 종료**일 수도 **붙잡아달라**일 수도.  
  - 이 챕터에서 **조건부 대사**(플래그/변수에 따라 동일 문장이 다르게 들리는 장치) 강하게 사용.
- **클라이맥스 선택**:
  - 사과 타이밍(현장/후/다음날), 사과 방식(설명/책임/요청)
- **로그 테마**: “지금은 말이 아니라 안전감” / “설명은 때로 변명으로 들림”

### Ch5. 결말: 말하지 않은 문장들
- **시간/장소**: 비 오는 날 카페(또는 캠퍼스 벤치), 대화의 마무리
- **핵심 갈등**: “정답 맞히기”를 멈추고, 서로의 방식(직설/완곡)을 합의할 수 있는지.
- **엔딩 분기**: 변수/플래그 기반 4~6개(아래 4절 스펙 참조)
- **로그 테마**: 밈의 해석은 도구일 뿐 / 관계는 누적된 행동으로 결정

## 3) 변수 밸런싱 설계 (범위/예산/검증)

### 3.1 변수 범위(권장)
- `trust`: 0~10
- `sense`: 0~10
- `interest`: 0~10
- `fatigue`: 0~10 (높을수록 나쁨)
- 모든 변수는 **클램프(clamp)** 처리: 0 미만/10 초과 방지

### 3.2 챕터별 “변수 예산”(권장)
> 5챕터 동안 엔딩 조건에 도달 가능하도록, 평균적으로 **한 챕터당 1~2 정도의 상승 여지**가 있어야 합니다.

- Ch1: trust +0~+2, sense +0~+2, interest +0~+1, fatigue +0~+2  
- Ch2: trust +0~+2, sense +0~+2, interest +0~+2, fatigue +0~+2  
- Ch3: trust +0~+2, sense +0~+2, interest +0~+2, fatigue +0~+3  
- Ch4: trust -2~+2(진폭 크게), sense +0~+2, interest -2~+2, fatigue +0~+3  
- Ch5: 최종 보정(대화 성사 시 trust/sense +1, 번아웃 시 fatigue +2 등)

### 3.3 interest를 “행동”으로 올리는 규칙
- “관심은 해석이 아니라 **기억/행동/일관성**으로 상승”
- 예시 선택지 효과:
  - 약속 지키기/먼저 연락/구체적 제안 → `interest +1`, `trust +1`
  - 과도한 캐묻기/설명 강요 → `fatigue +1~2`, `interest -1` 가능

### 3.4 밸런스 검증(간단 규칙)
- 베스트 루트(최적 선택)로:
  - `trust`와 `sense`를 각각 **초기 3 → 7 이상** 도달 가능해야 함(총 +4 필요)
- 워스트 루트로:
  - `fatigue`가 **8 이상** 도달 가능해야 “번아웃 엔딩”이 활성화됨

## 4) 엔딩 스펙 (우선순위/에필로그 포맷)

### 4.1 우선순위 규칙(명시)
- `endings` 배열은 **위에서부터 순서대로 평가**한다.
- `when` 조건이 true인 **첫 번째 엔딩**이 적용된다.
- 따라서 `E_NEUTRAL: true`는 **배열 최하단** 고정.

### 4.2 엔딩 데이터 포맷 추가(권장)
`data/game.config.json`의 endings 항목에 `epilogue`를 추가:
```json
{
  "id": "E_SOLID",
  "title": "솔직해진 관계",
  "when": "vars.trust>=7 && vars.sense>=7 && vars.fatigue<8",
  "epilogue": {
    "bg": "rain_cafe",
    "lines": [
      {"who":"미나","text":"나 사실, 괜찮다는 말을 너무 쉽게 써버려."},
      {"who":"나","text":"그 말이 나한텐 신호였다는 걸 이제는 알아."}
    ],
    "summary": "서로의 언어를 합의했다. 정답이 아니라 규칙을 만들었다."
  }
}
```

### 4.3 엔딩 화면 UI 요구
- 엔딩 타이틀
- 에필로그 대사(3~8줄)
- 변수 요약(최종 trust/sense/interest/fatigue)
- 해금 로그 보기 버튼
- 리플레이(챕터 선택/리셋) 버튼

## 5) 노드 타입 확장 스펙 (setchar / playBgm / 조건부)

### 5.1 `setchar` 포맷(샘플 포함)
```json
{ "type":"setchar", "slot":"L", "char":"mina", "pose":"neutral", "expr":"smile", "next":"n10" }
```
- `slot`: "L" | "C" | "R" (좌/중/우)
- `char`: 캐릭터 id(예: "mina")
- `pose`: 전신/반신 포즈 키워드(에셋 없으면 문자열만 표시)
- `expr`: 표정 키워드(에셋 없으면 문자열만 표시)
- 엔진은 화면에 “Mina (neutral/smile)” 같은 텍스트 오버레이로 대체 가능

### 5.2 배경 전환(setbg) 연출 옵션
```json
{ "type":"setbg", "bg":"cafe_day", "transition":"fade", "duration":250, "next":"n2" }
```
- `transition`: "cut" | "fade"
- `duration`: ms

### 5.3 사운드 노드 타입(미리 정의)
```json
{ "type":"playBgm", "track":"bgm_calm_01", "fadeIn":800, "next":"n5" }
{ "type":"stopBgm", "fadeOut":500, "next":"n6" }
{ "type":"playSfx", "sfx":"msg_pop", "next":"n7" }
```
- 사운드 파일이 없으면 더미로 콘솔 로그만 찍어도 됨(훗날 붙이기 쉬움)

## 6) 조건부 분기/조건부 텍스트 포맷 (필수 보강)

### 6.1 조건 표현식(권장 방식: 작은 DSL)
노드에 `if` 배열을 두고, 첫 매칭 블록을 사용:
```json
{
  "type":"say",
  "who":"미나",
  "if":[
    {"when":"flags.includes('dodged_once') && vars.sense<5", "text":"아니야. 그냥… 너답다.", "next":"n21"},
    {"when":"vars.sense>=6", "text":"괜찮아. 근데 다음엔 미리 말해줘.", "next":"n21"},
    {"when":"true", "text":"아니야. 그냥 그렇다고.", "next":"n21"}
  ]
}
```
- 장점: altText/condition 난립 방지, 확장 용이
- 엔진 구현: `when` 문자열을 안전하게 평가(권장: 제한된 파서)  
  - 빠른 MVP는 `Function("vars","flags","visited","return ("+when+")")` 가능  
  - 단, **외부 입력이 없고 로컬 JSON만** 읽는다는 전제에서만 사용

### 6.2 조건부 next(선택지에서도 동일)
```json
{
  "type":"choice",
  "prompt":"(뭐라고 할까?)",
  "choices":[
    {
      "label":"설명으로 밀어붙인다",
      "effects":[{"op":"add","var":"fatigue","val":2}],
      "nextIf":[
        {"when":"vars.trust>=6", "next":"nA"},
        {"when":"true", "next":"nB"}
      ]
    }
  ]
}
```

### 6.3 분기 합류 설계 규칙(스토리/리플레이감)
- “바로 합류”만 쓰지 말고, 분기 뒤에 **1~3노드 정도의 후속 결과**를 둔다.
- 합류 지점에서 같은 대사를 쓰더라도 위의 `if`로 **톤/의미를 변화**시킨다.
- 챕터당 최소:
  - 조건부 say 4개 이상
  - 조건부 next 2개 이상

## 7) 로그 해금 조건 포맷 확장(AND/OR/변수)

### 7.1 unlock 스키마를 통일: `when`으로 확장
```json
{
  "unlock": { "when":"visited.n3 || flags.includes('dodged_once') || vars.sense>=5" },
  "title": "...",
  "text": "..."
}
```
- 엔진이 유지하는 `visited`는 `{[nodeId]: true}` 형태로 접근 가능하게 제공  
  - 예: `visited.n3 === true`

### 7.2 복합 조건(예)
- “Ch2에서 특정 선택 + trust>=5”
```json
{ "unlock": { "when":"visited.c2_n14 && vars.trust>=5" }, "title":"...", "text":"..." }
```

## 8) UI/UX 결정(스펙으로 고정)

### 8.1 텍스트 표시
- 기본: **한 번에 표시(즉시)**  
- 옵션: 설정에서 “타이핑 효과” 토글 + 속도 슬라이더(예: 0~60ms/char)

### 8.2 선택지 시간 제한
- MVP: 없음  
- 확장(선택): Ch4 갈등 파트에서만 “침묵 선택지(타이머)” 가능  
  - 타이머 8초, 시간 초과 시 자동 선택(대체 선택지)

### 8.3 전환 연출
- 배경: fade 250ms 기본
- 캐릭터: cut(즉시) 기본, 나중에 slide-in 추가 가능

### 8.4 모바일/반응형
- 필수: 360px 너비에서 대사창/버튼이 깨지지 않게  
- 버튼은 세로 스택, 폰트 최소 16px

### 8.5 접근성/조작
- 키보드: Space(다음), 1~4(선택지), Esc(메뉴)
- 폰트 크기 옵션: S/M/L(예: 16/18/20)

## 10) 세이브 데이터 포맷(명세)

### 10.1 저장 슬롯 구조(예)
```json
{
  "version": 1,
  "savedAt": "2026-01-28T12:34:56.000Z",
  "state": {
    "chapter": "ch3",
    "node": "n12",
    "vars": {"trust":6,"sense":5,"fatigue":4,"interest":6},
    "flags": ["check_later","apologized_fast"],
    "visited": {"ch1":{"n3":true,"n10":true},"ch2":{"n7":true},"ch3":{"n1":true}},
    "unlockedLogs": {"ch1":[0,1,3],"ch2":[0,2]}
  },
  "settings": {"typing":false,"typingSpeed":20,"fontSize":"M"}
}
```
- `visited`는 챕터 단위로 분리 권장(충돌 방지)
- `unlockedLogs`는 “인덱스 배열”로 저장(로그 파일 entries 순번)

### 10.2 버전 마이그레이션
- `version` 증가 시:
  - 없으면 기본값 주입
  - 에러 시 “세이브 불러오기 실패” 안내 + 초기화 옵션 제공

## 11) 테스트/QA + 디버그 모드(권장)

### 11.1 시나리오 정적 검증(자동)
엔진 없이도 JSON만 검사하는 스크립트(`tools/validate_scenario.js`) 계획:
- 모든 `next`/`start`가 존재하는 노드를 가리키는지
- 도달 불가능 노드 탐지(BFS)
- 무한 루프(선택지 없이 순환) 탐지
- 사용되지 않는 플래그/변수 탐지(오탈자)
- endings `when` 파싱 가능 여부

**Prompt (Codex, QA 도구 추가):**
- `girlanguage_vn/tools/validate_scenario.js` 작성
- Node.js로 실행(외부 패키지 없이)
- `node tools/validate_scenario.js` 실행 시:
  - 각 챕터별 오류/경고 리포트 출력
  - exit code: 오류 있으면 1, 없으면 0

### 11.2 런타임 디버그 모드
- URL 파라미터 `?debug=1`이면 HUD에 디버그 패널 표시:
  - 변수 슬라이더(0~10)
  - 플래그 추가/삭제
  - 노드 ID 입력 후 jump
  - 로그 강제 해금 토글
